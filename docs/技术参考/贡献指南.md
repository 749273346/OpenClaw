# 惠电宝 - 开发规范与贡献指南 (Development & Contribution Guide)

## 1. 简介 (Introduction)
欢迎参与惠电宝（Huidianbao）的开发！本项目旨在通过开源协作，打造最智能的铁路电力安全助手。本指南定义了代码风格、提交规范、分支管理及测试要求。

## 2. 代码风格 (Code Style)

### 2.1 Python
- **PEP 8**: 遵循 PEP 8 编码规范。
- **Linting**: 使用 `flake8` 进行代码检查。
- **Formatting**: 使用 `black` 进行代码格式化。
- **Type Hints**: 鼓励使用类型提示（Type Hints），特别是在核心逻辑函数中。
- **Imports**: 按照标准库、第三方库、本地模块的顺序导入，每组之间空一行。

### 2.2 TypeScript / JavaScript
- **ESLint**: 遵循 ESLint (`airbnb` 或 `standard` 配置)。
- **Prettier**: 使用 Prettier 进行代码格式化。
- **Naming**: 变量名采用 `camelCase`，类名采用 `PascalCase`，常量采用 `UPPER_SNAKE_CASE`。

### 2.3 Markdown
- **Headers**: 使用 `#` 表示标题层级，避免跳级。
- **Lists**: 使用 `-` 表示无序列表，数字表示有序列表。
- **Code Blocks**: 代码块必须指定语言（如 `python`）。

## 3. 分支管理 (Branch Management)
- **main**: 主分支，始终保持可部署状态。仅接受来自 `develop` 或 `hotfix` 分支的合并。
- **develop**: 开发分支，用于集成新功能。
- **feature/**: 功能分支，从 `develop` 分支检出，命名格式 `feature/<feature-name>`。
- **bugfix/**: 修复分支，从 `develop` 分支检出，命名格式 `bugfix/<issue-id>`。
- **hotfix/**: 紧急修复分支，从 `main` 分支检出，修复后合并回 `main` 和 `develop`。

## 4. 提交规范 (Commit Message Convention)
遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：
- `feat`: 新增功能 (Feature)
- `fix`: 修复 Bug (Bug Fix)
- `docs`: 文档变更 (Documentation)
- `style`: 代码格式调整 (Formatting, missing semi colons, etc)
- `refactor`: 代码重构 (Refactoring)
- `test`: 增加测试 (Adding missing tests)
- `chore`: 构建过程或辅助工具变动 (Build process, auxiliary tools)

示例: `feat(rag): add hybrid search support with BM25 and Embedding`

## 5. 开发流程 (Development Workflow)
1. **Fork & Clone**: Fork 本仓库并克隆到本地。
2. **Create Branch**: 创建功能分支。
3. **Develop**: 编写代码并进行本地测试。
4. **Commit**: 提交代码，确保 Commit Message 清晰。
5. **Push**: 推送分支到远程仓库。
6. **Pull Request**: 发起 PR 到 `develop` 分支，描述变更内容与测试结果。
7. **Code Review**: 等待 Maintainer 审核代码，根据反馈进行修改。
8. **Merge**: 审核通过后合并代码。

## 6. 测试要求 (Testing Requirements)
- **Unit Tests**: 核心逻辑（如检索算法、意图分类）必须包含单元测试。
- **Integration Tests**: 涉及外部 API（如 DeepSeek, Zhipu AI）的模块需提供 Mock 测试或沙盒测试脚本。
- **Manual Verification**: 对于 UI/交互类变更，需提供测试截图或录屏。
